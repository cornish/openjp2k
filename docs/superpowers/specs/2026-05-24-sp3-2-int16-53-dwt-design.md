# SP3.2 — Int16 5/3 DWT Path for ≤8-bit Precision

## Goal

Add an int16 path to the 5/3 lossless inverse DWT (`opj_dwt_decode_tile`)
for components with input precision ≤ 8 bits. Targets DICOM 8-bit
(ultrasound, secondary capture, some others) and photographic 8-bit
corpora. AVX2 only, gated by `OPJ_ENABLE_AVX2`.

The lift kernels themselves run ~2× faster from doubled vector lane count
(16 int16 lanes per `__m256i` vs 8 int32 lanes). Predicted wall-clock win:
1.5-3% on ≤8-bit lossless workloads. Conversion to/from int16 amortizes
across all decomposition levels.

This is **SP3.2** — the second deliverable in the SP3 (IDWT polish)
decomposition. SP3.1 shipped the AVX2 9/7 lossy lift; SP3.2 is the 5/3
lossless analogue for narrow-precision data.

## Background

The 5/3 integer DWT in `src/lib/openjp2/dwt.c` currently has an
`__AVX2__`-gated path that operates on `__m256i` containing 8 int32
elements per vector. For data with input precision ≤ 8 bits, the
intermediate lifting values cannot exceed int16 range, so we can pack
two int32 elements into one int16 slot and process 16 elements per
vector — half the cycles for the same data.

DICOM 8-bit components (CIE J2KR_US, J2KR_SC) and photographic 8-bit
content (JP2 photo, web imagery, WSI tiles at 8-bit) are the primary
consumers.

## Design

### Activation

At `opj_dwt_decode_tile` entry, inspect the source image component's
precision:

```c
const opj_image_comp_t *image_comp = &p_tcd->image->comps[compno];
const OPJ_BOOL use_int16 = (image_comp->prec <= 8);
#ifdef __AVX2__
    if (use_int16) {
        return opj_dwt_decode_tile_int16(tp, tilec, numres);
    }
#endif
/* existing int32 path */
```

The check is per-tilec, so mixed-precision multicomponent images (rare
but legal: e.g., 8-bit RGB + 16-bit alpha) dispatch each component
independently.

Signedness does not enter the gate: encoders DC-level-shift unsigned
data by `-2^(prec-1)` before encoding, so dequantized coefficients are
always signed regardless of `image_comp.sgnd`. The range bound is on
magnitude, not on sign.

### Data flow (int16 path)

```
opj_dwt_decode_tile_int16(tilec) {
    int16_t* scratch = aligned_alloc(tilec_w * tilec_h * sizeof(int16_t));
    pack_int32_to_int16(scratch, tilec->data, tilec_w * tilec_h);

    for (resno = 1; resno < numres; ++resno) {
        for each row:
            opj_idwt53_h_int16(row);
        for each col-stripe:
            opj_idwt53_v_int16_mcols(col);
    }

    unpack_int16_to_int32(tilec->data, scratch, tilec_w * tilec_h);
    aligned_free(scratch);
}
```

Scratch buffer is allocated per tilec (or pooled — see Risks #5). The
pack converts `_mm256i` of 8 int32s → 128-bit lane of 8 int16s via
`_mm256_packs_epi32` (saturating; for prec ≤ 8 data no value exceeds
int16 range, so saturation never fires — assert in Debug). The unpack
uses `_mm256_cvtepi16_epi32` (sign-extend).

### New kernels

Four mirrored AVX2 functions in `dwt.c`, modeled after the existing int32
AVX2 kernels:

| New (int16) | Mirrors (int32) |
|---|---|
| `opj_idwt53_h_cas0_int16_avx2` | `opj_idwt53_h_cas0` AVX2 branch |
| `opj_idwt53_h_cas1_int16_avx2` | `opj_idwt53_h_cas1` AVX2 branch |
| `opj_idwt53_v_cas0_mcols_int16_avx2` | `opj_idwt53_v_cas0_mcols_SIMD` AVX2 branch |
| `opj_idwt53_v_cas1_mcols_int16_avx2` | `opj_idwt53_v_cas1_mcols_SIMD` AVX2 branch |

Plus two small helper functions:

- `opj_dwt_pack_int32_to_int16` — converts a buffer (AVX2-vectorized via
  `_mm256_packs_epi32` with intra-lane fixup using
  `_mm256_permute4x64_epi64` to repair the per-lane packing).
- `opj_dwt_unpack_int16_to_int32` — converts back via
  `_mm256_cvtepi16_epi32`.

The lift formulas translate mechanically: `_mm256_add_epi32` →
`_mm256_add_epi16`, `_mm256_srai_epi32` → `_mm256_srai_epi16`,
`_mm256_sub_epi32` → `_mm256_sub_epi16`. No FMA, no rounding subtlety —
5/3 is integer arithmetic with shifts.

`PARALLEL_COLS_53` and `VREG_INT_COUNT` remain as defined for the int32
path; the int16 path introduces its own `PARALLEL_COLS_53_INT16` = 32
(2 × the int16 vector lane count of 16) for the vertical mcols loop.

### Verification

- Conformance: `OPJ_ENABLE_AVX2=ON` build must show the same 8 pre-existing
  NR-DEC-md5 failures, no new mismatches on any ≤8-bit corpus file.
- Diff-test: 90/0 on the smoke corpus (legacy vs fast at byte level).
- Manual byte-cmp: pick three 8-bit files (one DICOM US, one photo,
  one WSI tile), decode with `OPJ_ENABLE_AVX2=ON` and `=OFF`, byte-cmp
  the raw decoded planes.

### Bench gate

| Corpus | n | Pass | Target |
|---|---|---|---|
| DICOM J2KR 8-bit (US/SC subset of medical corpus) | ~5-8 | ≥ +1% gmean | ≥ +2% |
| Smoke 8-bit lossless subset | ~10-15 | ≥ +1% gmean | ≥ +2% |
| WSI tile 8-bit sample | ~5 | ≥ +1% gmean | ≥ +2% |
| Smoke overall | 90 | within ±1% gmean | — |
| Archival (lossy 9/7, ≥8-bit) | 2 | within ±1% gmean | — |

The archival files are 9/7 lossy — SP3.2 doesn't touch their path, so
the bench measures noise level. If archival regresses by ≥1%, that's an
unintended side effect (e.g., the new code being compiled in changed
icache pressure) and must be investigated.

## Files Modified

- `src/lib/openjp2/dwt.c`:
  - Add 4 new AVX2 int16 kernels (`opj_idwt53_h_cas{0,1}_int16_avx2`,
    `opj_idwt53_v_cas{0,1}_mcols_int16_avx2`).
  - Add 2 conversion helpers (pack/unpack int32↔int16).
  - Add `opj_idwt53_h_int16` and `opj_idwt53_v_int16` dispatchers that
    call the cas0/cas1 int16 kernels (mirror the existing
    `opj_idwt53_h` / `opj_idwt53_v` dispatchers).
  - Add `opj_dwt_decode_tile_int16` orchestrator that owns the int16
    scratch buffer, submits int16 thread-pool jobs (mirror
    `decode_h_job` / `decode_v_job`), and calls the int16 dispatchers.
  - Add the prec ≤ 8 dispatch check at `opj_dwt_decode_tile` entry that
    forwards to `opj_dwt_decode_tile_int16` when active. The int16
    orchestrator owns the entire decode for that tilec — no shared
    code with the int32 orchestrator inside the IDWT loop.
  - Estimated net add: ~400 LOC.

- No other files modified. The dispatch is internal to dwt.c.

## Scope Boundaries

- **5/3 lossless only.** SP3.1 covered the 9/7 lossy lift; SP3.2 is the
  integer analogue. Different code path, different math.
- **prec ≤ 8 only.** Strict gate. 9-bit and above stays on the int32
  path. (See Open Questions for the 12-bit deferral story.)
- **AVX2 only.** Non-AVX2 builds (no `OPJ_ENABLE_AVX2` or no compiler
  AVX2 support) run the existing int32 path. SSE2 int16 deferred.
- **Full-tile only.** `opj_dwt_decode_partial_tile` (for ROI/zoom)
  stays on int32. SP3.2b would add the partial-path int16.
- **No new build options.** Activates automatically inside the existing
  `OPJ_ENABLE_AVX2` envelope.

## Risks

1. **Overflow on ≥9-bit data.** Strict gate at prec ≤ 8 prevents this.
   Defense-in-depth: Debug assertion inside `pack_int32_to_int16` that
   no source element exceeds int16 range (`_mm256_packs_epi32`
   saturates so we'd need a sentinel check).

2. **Conversion overhead dominates on small tiles.** The pack + unpack
   are O(N) over the whole tilec area. For tiny tiles (e.g., DICOM
   thumbnail), the conversion can eat the win. Mitigation: bench-measure;
   if measurable regression on small tiles, add a `tilec_w * tilec_h ≥ T`
   guard (T to be determined empirically).

3. **Per-tilec scratch alloc churn.** Each tilec allocates a scratch
   buffer at decode start, frees at end. For images with many small
   tilecs this adds malloc overhead. Mitigation: reuse the D6.1 TCD
   buffer pool (`opj_tcd_pool_alloc_data`) for the scratch buffer.

4. **`_mm256_packs_epi32` cross-lane semantics.** The intrinsic packs
   within each 128-bit lane separately, so the result has elements in
   `[0,1,2,3, 8,9,10,11, 4,5,6,7, 12,13,14,15]` order from input. Fix
   with `_mm256_permute4x64_epi64(_, 0xD8)` to restore linear order.
   Trivial, but must be tested.

5. **Code duplication.** The new int16 kernels mirror the int32 ones
   line-for-line with type-name changes. Macroing for type
   parameterization is possible but is a separate refactor. Accept the
   duplication for SP3.2; refactor later if both branches keep
   diverging.

## Open Questions

None for SP3.2 itself. The 12-bit extension (DICOM CT/MR workhorse)
deserves its own deliverable (SP3.2c) once SP3.2 lands and we can
measure the actual win on the narrower-precision corpus. The 12-bit
math is borderline for int16 (intermediate values can reach ~±2^16
through 5 decomposition levels), so 12-bit would need either careful
saturation analysis or a defensive narrowing to int16 with overflow
detection.

## Decision Gate

After implementation:
- **DICOM 8-bit subset speedup**: ≥ +1% (pass), ≥ +2% (target).
- **Photographic 8-bit / WSI tile speedup**: ≥ +1% (pass), ≥ +2%
  (target).
- **No smoke or archival regression** beyond ±1%.
- **Conformance**: 8 pre-existing failures only under
  `OPJ_ENABLE_AVX2=ON`.
- **Build**: passes with `OPJ_ENABLE_AVX2=ON` and `OPJ_ENABLE_AVX2=OFF`.

If DICOM 8-bit improves < 0.5%, investigate before tagging. If smoke
regresses > 1%, revert.
