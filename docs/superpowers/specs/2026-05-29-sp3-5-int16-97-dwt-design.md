# SP-3.5 — Int16 9/7 DWT Path for ≤8-bit Precision (Lossy)

## Goal

Add an int16 fixed-point path to the inverse 9/7 (irreversible / lossy)
DWT in `src/lib/openjp2/dwt.c` for components with input precision ≤ 8
bits. Targets the largest unaddressed lossy-decode opportunity after D8:
the SP-2.1a post-D8 profile localized 13% of single-thread runtime on
1024² 8-bit lossy WSI tiles to the float-AVX2 9/7 inverse DWT (vs grok's
5.6% in the same role using their 16-bit Highway implementation).

Halves SIMD lane width (8 float lanes → 16 int16 lanes per `__m256i`) →
2× throughput on the dominant kernel. Predicted: WSI 1024² 8-bit lossy
moves from current +3.33% vs grok at t=8 to ~+5-6% vs grok; 8-bit lossy
iter slice gains +5-8% on top of D8.

This is **SP-3.5** — the 9/7 analogue of SP3.2's 5/3 int16 work. AVX2
only, gated by `OPJ_ENABLE_AVX2`. Departs from the project's byte-exact
SIMD pattern: 9/7 lift coefficients are real-valued, so a Q-format
fixed-point implementation introduces ±1 LSB rounding deltas vs the
float reference. Acceptable for lossy decode per the JP2K spec and
matching grok's approach.

## Background

The inverse 9/7 (irreversible) DWT in `dwt.c` currently has an
`__AVX2__`-gated float-lift path from SP3.1 (commit `c1df838c`). It
operates on `__m256` containing 8 float32 elements per vector, applying
the lift formulas:

```
s'[i]  = s[i] + δ × (d[i-1] + d[i])         /* update 2 */
d'[i]  = d[i] + γ × (s'[i] + s'[i+1])       /* predict 2 */
s''[i] = s'[i] + β × (d'[i-1] + d'[i])      /* update 1 */
d''[i] = d'[i] + α × (s''[i] + s''[i+1])    /* predict 1 */
out_low[i]  = K   × s''[i]                  /* scale */
out_high[i] = 1/K × d''[i]
```

with coefficients α=-1.586134342, β=-0.052980118, γ=0.882911075,
δ=0.443506852, K=1.230174105, 1/K=0.812893066.

For 8-bit precision data, intermediate coefficient magnitudes through
all 5 decomposition levels stay within ±32 (verified by range analysis;
see Risks #1). This fits in a **Q5.11 fixed-point format** (5 integer
bits, 11 fractional bits, signed int16): integer range ±32, fractional
resolution ~5e-4 — adequate precision for 8-bit-input lossy output.

Grok's profile (post-D8, on the same 1024² WSI tile) shows their DWT
running through `hwy_h_synth_16_97` (3.66%) + `hwy_v_synth_16_97`
(1.95%) = 5.6% of decode time. Our equivalent path (`opj_dwt_decode_real`
+ `opj_v8dwt_decode` + `opj_v8dwt_interleave_h` + `opj_dwt97_decode_v_func`
+ `opj_dwt97_decode_h_func`) consumes ~13% of decode time. Closing this
gap puts our 1024² 8-bit lossy timing ~1 ms below grok at single-thread
on the WSI test file.

## Design

### Activation

At `opj_dwt_decode_real` entry, gate on the source component's
precision and the `whole_tile_decoding` flag, mirroring SP3.2's
activation pattern:

```c
OPJ_BOOL opj_dwt_decode_real(opj_tcd_t *p_tcd, opj_tcd_tilecomp_t* tilec,
                             OPJ_UINT32 numres)
{
    if (p_tcd->whole_tile_decoding) {
#ifdef __AVX2__
        const OPJ_UINT32 compno = (OPJ_UINT32)(tilec -
                                  p_tcd->tcd_image->tiles->comps);
        const opj_image_comp_t *image_comp = &p_tcd->image->comps[compno];
        if (image_comp->prec <= 8) {
            return opj_dwt_decode_tile_real_int16(p_tcd->thread_pool,
                                                   tilec, numres);
        }
#endif
        return opj_dwt_decode_tile_real(p_tcd->thread_pool, tilec, numres);
    } else {
        return opj_dwt_decode_partial_tile_real(tilec, numres);  /* unchanged */
    }
}
```

Per-tilec check matches SP3.2; mixed-precision multicomponent images
dispatch each component independently.

### Q5.11 fixed-point format

**Format:** signed int16, 5 integer bits (range ±32), 11 fractional bits
(resolution 1/2048 ≈ 4.88e-4).

**Pre-DWT pack:** scale `tilec->data` (int32, dequantized 9/7
coefficients) by `2^11` and pack into int16 with saturation.

```c
/* Per element: int32 in [-32, +32) -> int16 in [-65536, +65535) -> packed int16 */
int16 = (int16)CLAMP((int32_value << 11), INT16_MIN, INT16_MAX);
```

Range analysis (Risks #1): on 8-bit input data, dequantized 9/7
coefficients fit in ±32 with margin. Intermediate lift values stay
within ±32 across all 5 decomposition levels.

**Post-DWT unpack:** shift right by 11 (with rounding), sign-extend to
int32, store to `tilec->data`.

```c
int32 = ((int32)int16_value + (1 << 10)) >> 11;
```

### Lift coefficient constants

Q1.15 representation (suitable for `_mm256_mulhrs_epi16` rounding
multiply, which computes `(a × b + 0x4000) >> 15`):

| Coefficient | Float value | Q1.15 int16 | Hex |
|---|---:|---:|---|
| α  | −1.586134342 | (out of range) | use Q2.14 = -25996 |
| β  | −0.052980118 | -1736 | 0xF938 |
| γ  | +0.882911075 | +28932 | 0x7104 |
| δ  | +0.443506852 | +14533 | 0x38C5 |
| K  | +1.230174105 | (out of range) | use Q2.14 = +20157 |
| 1/K| +0.812893066 | +26637 | 0x680D |

α and K exceed Q1.15 range (≥1.0). Two options: shift to Q2.14 (range
±2) at the cost of one bit fractional precision, OR split the multiply
into integer + fractional parts. Recommend **Q2.14 for α and K, Q1.15
for the rest** — simpler, accepted ~5e-4 → ~1e-3 precision loss on those
multiplies. Range analysis confirms intermediate values still fit in
int16 with margin.

### New kernels in dwt.c (~600-800 LOC)

Modeled after the existing SP3.1 AVX2 9/7 lift kernels:

| New (int16 9/7) | Mirrors (existing float 9/7) |
|---|---|
| `opj_v16dwt_decode_step1_int16_avx2` | `opj_v8dwt_decode_step1` (AVX2 branch) |
| `opj_v16dwt_decode_step2_int16_avx2` | `opj_v8dwt_decode_step2` (AVX2 branch) |
| `opj_v16dwt_interleave_h_int16_avx2` | `opj_v8dwt_interleave_h` (AVX2 branch) |
| `opj_v16dwt_interleave_v_int16_avx2` | `opj_v8dwt_interleave_v` (AVX2 branch) |
| `opj_dwt97_decode_h_func_int16` | `opj_dwt97_decode_h_func` |
| `opj_dwt97_decode_v_func_int16` | `opj_dwt97_decode_v_func` |
| `opj_dwt_decode_tile_real_int16` | `opj_dwt_decode_tile_real` |

Plus two helpers:

- `opj_dwt_pack_int32_to_int16_q511` — int32 input → int16 Q5.11, AVX2
  vectorized (`_mm256_packs_epi32` with cross-lane fixup via
  `_mm256_permute4x64_epi64(_, 0xD8)`, pre-shift left by 11 with
  saturation).
- `opj_dwt_unpack_int16_q511_to_int32` — int16 Q5.11 → int32, AVX2
  vectorized (`_mm256_cvtepi16_epi32` + arithmetic shift right by 11
  with rounding).

**Lift multiply uses `_mm256_mulhrs_epi16`** — single instruction,
hardware-rounded Q1.15 multiply. For Q2.14 coefficients (α, K), shift
result left by 1 to compensate.

### Orchestrator data flow

```
opj_dwt_decode_tile_real_int16(tilec) {
    int16_t* scratch = aligned_alloc(tilec_w * tilec_h * sizeof(int16_t));
    opj_dwt_pack_int32_to_int16_q511(scratch, tilec->data, tilec_w * tilec_h);

    for (resno = 1; resno < numres; ++resno) {
        for each row:
            opj_dwt97_decode_h_func_int16(row);
        for each col-stripe:
            opj_dwt97_decode_v_func_int16(col);
    }

    opj_dwt_unpack_int16_q511_to_int32(tilec->data, scratch, tilec_w * tilec_h);
    aligned_free(scratch);
}
```

Scratch buffer is per-tilec (alloc/free at orchestrator boundaries).
Could be pooled via the D6.1 buffer pool as a follow-up; per-tilec alloc
is acceptable for SP-3.5.

### Files Modified

- `src/lib/openjp2/dwt.c`: new int16 9/7 kernels, helpers, orchestrator,
  and dispatch hook at `opj_dwt_decode_real` entry. ~600-800 LOC added.
- New PSNR verification test (see below): probably in `tests/unit/` or
  `tests/`. ~150-200 LOC.

No other source files modified.

## Verification

### Conformance fingerprint snapshot

Run `OPJ_ENABLE_AVX2=ON` conformance after SP-3.5 lands. Enumerate
every NR-DEC-md5 test whose pass/fail status differs from the pre-SP-3.5
baseline (the 8 pre-existing failures). Document each shift in the
outcome spec:
- Tests that newly fail at ±1 LSB rounding: expected for some 9/7 lossy
  files; non-blocking if the count is small (<5).
- Tests that newly pass: bonus; document.
- Test that previously passed and still passes: must match.

After the snapshot is documented, it becomes the new conformance
baseline for future deliverables.

### PSNR verification harness (new ctest target)

New ctest target `sp3_5_psnr_test`: 5 representative 8-bit lossy files
× decode with `OPJ_ENABLE_AVX2=ON` (int16 path) and `OPJ_ENABLE_AVX2=OFF`
(float reference). Compute PSNR (peak signal-to-noise ratio) per
component:

```
PSNR = 10 × log10(MAX_VAL² / MSE)
where MSE = mean of (out_int16 - out_float)²
```

Require **PSNR ≥ 60 dB** per component per file. 60 dB corresponds to
~max delta ~5 across 8-bit dynamic range — effectively numerically
indistinguishable to human perception or downstream processing.

Test files:
- `corpus/public/wsi-tiles/1024x1024/JP2K-33003-2__T1024_c041r117.j2k`
  (the SP-3.5 motivating WSI file)
- A 1024² synthetic 8-bit lossy from smoke
- An 8-bit photographic file from archival (if available; else a second
  synthetic)
- A 256² or 240² WSI tile from corpus/public/wsi-tiles/240x240/
- One 8-bit ultrasound DICOM J2KR file (`J2KR_US1_J2KR.j2k`) — exercises
  the J2KR decode path post-D8

Build the harness around the existing `test_decode_area` or
`test_tile_decoder` helpers; reuse `compare_raw_files` infrastructure
for the PSNR computation.

### Build / AVX2-OFF behavior

- `OPJ_ENABLE_AVX2=OFF`: build clean, conformance unchanged from main
  pre-SP-3.5. The int16 path code is `#ifdef __AVX2__` gated; AVX2-OFF
  builds the float path only.
- `OPJ_ENABLE_AVX2=ON`: build clean, conformance fingerprint shifts as
  documented above.

### Manual byte-cmp is NOT used

Explicit departure from D8/SP3.2 pattern. PSNR replaces byte-cmp because
float vs Q5.11 fixed-point are NOT byte-equivalent.

## Scope Boundaries

- **Lossy (9/7) only.** Lossless (5/3) path is SP3.2 territory; untouched.
- **prec ≤ 8 only.** 9-bit and above stays on the SP3.1 float-AVX2 path.
  SP-3.5b for higher precisions is a separate deliverable (likely needing
  Q1.15 plus careful saturation analysis, or wider Q-format).
- **Full-tile only.** Partial-tile / ROI 9/7 stays on float — SP3.2b's
  negative-result lesson applies. SP-3.5b-partial deferred.
- **AVX2 only.** Non-AVX2 builds run the existing SP3.1 float path.
  SSE2 int16 and NEON deferred. Highway runtime dispatch (SP-3.4 on the
  roadmap) is the right home for cross-target SIMD; SP-3.5 is the AVX2
  proof-point.
- **No new build option.** Activates inside the existing `OPJ_ENABLE_AVX2`
  envelope.
- **No T1 changes.** T1 fast paths are at parity with grok (per profile).

## Risks

1. **Q-format overflow on real codestreams.** Some 8-bit content may
   produce coefficients exceeding ±32 (Q5.11 integer range). Range
   analysis says no; empirical confirmation needed during implementation.
   Mitigation: saturating arithmetic everywhere (`_mm256_adds_epi16` /
   `_mm256_subs_epi16`). Saturation produces approximation, never UB.
   Initial implementation phase measures actual coefficient magnitudes
   on 20-30 representative 8-bit lossy files before locking Q5.11.

2. **PSNR threshold mis-set.** 60 dB is essentially "numerically
   indistinguishable" but accumulated Q-format rounding through 5 lift
   levels × 3 components × multiple multiplies per pixel could surprise.
   Mitigation: implementation phase first measures actual PSNR on 30+
   representative files; if any are <60 dB, investigate cause
   (Q-format tightening, per-pixel saturation pattern, or multiply
   ordering) before locking.

3. **Conformance fingerprint shift larger than expected.** Could be 5
   tests shift, could be 30. If many shift, hard to enumerate and
   justify each. Mitigation: if more than 10 tests shift, **pause and
   investigate before merging** — likely an implementation bug
   producing systematic error, not legitimate rounding drift.

4. **Implementation complexity higher than SP3.2.** SP3.2's 5/3 lift
   used only integer adds and shifts (no multiplies). SP-3.5's 9/7 has
   Q-format multiplies (6 distinct coefficients), Q2.14 vs Q1.15
   handling, scaling at the K / 1/K boundary, and saturation handling.
   Estimated 600-800 LOC vs SP3.2's ~970. Probably a 2-3 week
   deliverable. Mitigation: structure implementation into sub-steps
   (orchestrator + helpers → first lift kernel → remaining kernels →
   bench gate) and verify PSNR at each step.

5. **Partial-tile / ROI path stays on float.** SP3.2b's negative-result
   lesson applies. SP-3.5b (partial int16 9/7) is explicitly out of
   scope here, parked, and not pre-committed. Pursuing it would need
   profile data confirming a partial-decode 9/7 hot path that doesn't
   exist for SP3.2-style sparse arrays.

6. **Cumulative MT amplification may exceed predictions.** D8's MT
   scaling was super-linear (+6.42% t=1 → +9.96% t=8). If SP-3.5
   follows the same pattern, the predicted +5-8% iter t=8 may be
   conservative. Bonus if so; the predicted ranges in the bench gate
   are intentionally conservative.

7. **Cross-slice icache effects.** Adding ~700 LOC to dwt.c may shift
   the existing int32 9/7 float path's code layout. SP3.2 had a known
   +4% accidental cross-slice surprise; SP3.2b had a −1% accidental
   regression. Direction is unpredictable. Mitigation: measure smoke +
   iter overall at t=1 (where SIMD wins are smaller and icache is more
   visible); document residue in outcome.

8. **Highway dispatch (SP-3.4) might obsolete SP-3.5.** Highway provides
   portable SIMD across SSE2/AVX2/AVX-512/NEON/SVE. If SP-3.4 lands
   later and re-implements 9/7 via Highway with 16-bit lanes, SP-3.5's
   hand-rolled AVX2 path becomes redundant on x86. Acceptable: SP-3.5
   is the AVX2 proof-point and immediate win; SP-3.4 (when it happens)
   can replace SP-3.5 with a portable equivalent.

## Open Questions

1. **Q-format final choice (Q5.11 vs alternative).** Q5.11 is the
   starting recommendation based on range analysis. Implementation phase
   measures actual coefficient ranges across a representative 30-file
   sample. If margins are uncomfortably tight, drop to Q4.12 (more
   integer headroom, less precision); if margins are abundant, consider
   Q6.10 (more precision). Decide before the first kernel commits.

2. **Should the scratch buffer route through the D6.1 buffer pool?**
   Per-tilec `aligned_alloc` is the simplest implementation. Pool
   integration is a perf win on heavy multi-tile workloads (WSI
   pipelines, archival batches). Defer to a follow-up if measurement
   shows alloc churn.

3. **Test files for PSNR harness.** Need to pick 5 files at spec time;
   the WSI tile + smoke synthetic are obvious, the photographic /
   archival / DICOM ones depend on what's actually 8-bit in the corpus.
   Resolve at implementation start.

## Decision Gate

After implementation:

- **WSI 1024² 8-bit lossy iter t=1 / t=8**: ≥ +3% pass / ≥ +5% target.
- **8-bit lossy iter t=8**: ≥ +5% pass / ≥ +8% target.
- **No 12/16-bit lossy regression** beyond ±0.5%.
- **No lossless regression** beyond ±0.5%.
- **PSNR**: ≥ 60 dB on all 5 test files, all components.
- **Conformance**: fingerprint snapshot documented; ≤ 10 shifts;
  no systematic newly-failing pattern that suggests an
  implementation bug.
- **Build**: passes with `OPJ_ENABLE_AVX2=ON` and `OPJ_ENABLE_AVX2=OFF`.

If 8-bit lossy iter t=8 improves < +3%, the dispatch isn't engaging
or the SIMD helpers have hidden overhead — investigate before tagging.
If any PSNR test fails (< 60 dB), the Q-format or saturation pattern
is wrong — fix before benching. If the conformance fingerprint shifts
> 10 tests, pause and audit before merging.

## Cumulative project framing

SP-3.5 is the natural successor to D8: another profile-localized
per-stage SIMD win, targeting the next-largest stage gap (DWT 9/7)
after D8 closed the DC level shift gap. Predicted to close most of
the remaining grok lead on 8-bit lossy WSI workloads:

| Metric | Post-D8 (current) | **Post-SP-3.5 (predicted)** |
|---|---:|---:|
| WSI 1024² 8-bit lossy t=8 vs grok | +3.33% (ahead) | **+5-6% (further ahead)** |
| 8-bit lossy iter t=8 vs grok | −17.16% (behind) | −9 to −13% (smaller gap) |
| Iter overall t=8 vs grok | −4.00% (parity) | −2.5 to −3% (closer) |

Combined with D8, the project's competitive story on lossy 8-bit
workloads (WSI viewers, photographic web imagery, 8-bit ultrasound)
becomes strongly defensible — the SP3.x SIMD theme of "16 int16 lanes
> 8 float lanes" applied across the DWT path is the project's most
predictable lever post-D8.
